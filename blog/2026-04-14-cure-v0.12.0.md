# Cure v0.12.0: A Dependently-Typed Language for the BEAM

The BEAM virtual machine has Erlang. It has Elixir. It has Gleam and LFE and
Caramel and half a dozen others. What it has never had is a language that proves
your code correct at compile time using an SMT solver.

Until now.

## What is Cure?

Cure is a programming language that compiles to BEAM bytecode. It runs on the
same VM as your Erlang and Elixir code -- same OTP, same supervision trees,
same hot code loading. But it adds something those languages deliberately left
out: a type system that can reason about values, not just shapes.

```cure
type NonZero = {x: Int | x != 0}

fn safe_divide(a: Int, b: NonZero) -> Int = a / b
```

That `NonZero` is a refinement type. The constraint `x != 0` is not a runtime
check. It is a theorem. The compiler sends it to Z3, an SMT solver, and Z3
proves (or disproves) it at compile time. If you try to call `safe_divide(10, 0)`,
the compiler catches it before a single BEAM instruction executes.

## Why the BEAM?

The BEAM is the best runtime in existence for concurrent, fault-tolerant systems.
OTP supervisors, hot code upgrades, distribution -- nothing else comes close.
But its dynamically-typed nature means entire classes of bugs survive until
production. Dialyzer helps, but it's a success typings tool, not a proof system.

Cure keeps everything the BEAM gives you and adds compile-time guarantees on
top. A Cure module compiles to a regular `.beam` file. You can call it from
Erlang or Elixir. You can call Erlang or Elixir from Cure:

```cure
@extern(:erlang, :abs, 1)
fn abs(x: Int) -> Int
```

## The Syntax

Cure uses indentation structure. No `do`/`end`, no braces, no closing
delimiters. The visual layout is the structure:

```cure
mod MyApp.Math
  fn factorial(n: Int) -> Int
    | 0 -> 1
    | n -> n * factorial(n - 1)

  fn classify(x: Int) -> String
    | x when x > 0 -> "positive"
    | x when x < 0 -> "negative"
    | _             -> "zero"
```

Multi-clause functions use `|` for pattern dispatch. Guards use `when`.
Everything is an expression -- the last expression in a block is its value.

## First-Class Finite State Machines

This is where Cure gets interesting for anyone building protocols, workflows,
or embedded systems. FSMs are a language construct:

```cure
fsm TrafficLight
  Red    --timer-->     Green
  Green  --timer-->     Yellow
  Yellow --timer-->     Red
  *      --emergency--> Red
```

That is a complete, compilable file. The compiler generates an OTP `gen_statem`
module with `start_link`, `send_event`, and `get_state`. But before it
generates anything, it verifies:

- **Reachability**: every state is reachable from the initial state.
- **Deadlock freedom**: every non-terminal state has at least one outgoing transition.
- **Terminal validity**: declared terminal states exist in the transition graph.

The FSM runtime adds an ETS-backed registry, event history tracking, and batch
operations. You can spawn FSMs, name them, look them up, and monitor them --
all through a clean Cure API:

```cure
let pid = Std.Fsm.spawn(:"Cure.FSM.TrafficLight")
Std.Fsm.send(pid, :timer)
let state = Std.Fsm.state(pid)    # :green
```

## Protocols That Compile to Fast Dispatch

Cure's protocol system compiles to multi-clause guard-dispatched functions.
There is no runtime dictionary lookup, no vtable indirection:

```cure
proto Show(T)
  fn show(x: T) -> String

impl Show for Int
  fn show(x: Int) -> String = Std.String.from_int(x)

impl Show for Bool
  fn show(x: Bool) -> String = if x then "true" else "false"
```

The compiler generates:

```
show(X) when is_integer(X) -> show__for__int(X);
show(X) when is_boolean(X) -> show__for__bool(X).
```

Guard dispatch. No overhead. Bool is checked before Atom because the compiler
knows Erlang's type hierarchy and sorts clauses by specificity.

## The Type System

Cure's type checker is bidirectional. It infers types bottom-up and checks them
top-down. But the interesting parts are the extensions:

**Refinement types** constrain values with logical predicates verified by Z3:

```cure
type Positive = {x: Int | x > 0}
type Percentage = {p: Int | p >= 0 and p <= 100}
```

The compiler proves subtype relationships: `Positive <: NonZero` holds because
`forall x. x > 0 => x != 0`. It also catches impossible types:
`{x: Int | x > 0 and x < 0}` is unsatisfiable -- the compiler warns you.

**Pattern exhaustiveness** analysis covers Bool, Result, Option, List, and
any ADT. Miss a case, get a warning at compile time -- not a `function_clause`
crash at 3am.

**Guard-based flow typing** narrows parameter types inside guarded bodies. If
your function has `when x > 0`, then inside the body `x` has type
`{x: Int | x > 0}`, not just `Int`.

## Self-Hosted Standard Library

The standard library is written in Cure itself -- 12 modules, roughly 160
functions. It compiles with the same compiler that compiles user code:

- **Std.Core**: identity, compose, pipe, Result/Option types, boolean ops
- **Std.List**: map, filter, foldl, foldr, flat_map, zip_with, reverse, take, drop
- **Std.Math**: abs, sqrt, pow, pi, safe_div, clamp, sign
- **Std.String**: concat, split, trim, reverse, type conversions
- **Std.Show**: Show protocol with dispatch for all primitive types
- **Std.Eq/Std.Ord**: equality and ordering protocols
- **Std.Result**: dedicated Result module with map, and_then, or_else, flatten
- **Std.Fsm**: FSM runtime bridge (spawn, send, state, history)

Import resolution works at compile time. `use Std.List` makes `map` and
`filter` available as unqualified calls. Local functions always take priority
over imports.

## The Toolchain

Cure ships as a single escript binary with six subcommands:

```
cure compile examples/    # Compile .cure files to BEAM
cure run hello.cure       # Compile and execute (calls main/0)
cure check hello.cure     # Type-check only
cure lsp                  # Start the Language Server Protocol server
cure stdlib               # Compile the standard library
cure version              # Cure 0.12.0
```

The **LSP server** provides diagnostics on every keystroke, hover information
showing function signatures, keyword completions, and document symbols. It
communicates over stdio with Content-Length framing -- any LSP client works.

The **MCP server** exposes 7 tools for AI assistant integration: compile,
parse, type-check, analyze FSM, validate syntax, syntax help, and stdlib
documentation. Point Claude or Warp at `mix cure.mcp` and it can compile
your Cure code, explain errors, and suggest fixes.

The **optimizer** runs three passes between type checking and codegen:
constant folding (`2 + 3` becomes `5` at compile time), dead code elimination
(`if true then A else B` becomes `A`), and pipe chain inlining.

## Architecture

The compiler is built as a clean pipeline:

```
.cure source
  |  Lexer (indentation tracking, string interpolation)
  v
Token stream
  |  Pratt parser (precedence climbing, MetaAST 3-tuples)
  v
MetaAST
  |  Bidirectional type checker (refinement types, SMT, exhaustiveness)
  v
Typed MetaAST
  |  Optimizer (constant fold, dead code, pipe inline)
  v
Optimized MetaAST
  |  Codegen (Erlang abstract forms, protocol dispatch, import resolution)
  v
BEAM bytecode
```

Every stage emits structured events through a `Registry`-backed PubSub system.
The LSP, profiler, and MCP server all subscribe to these events. The profiler
can tell you exactly how many events each stage produced and how long
compilation took.

## Getting Started

```bash
git clone https://github.com/am-kantox/cure.git
cd cure
mix deps.get
mix test
mix escript.build
./cure run examples/recursion.cure
```

For editor support, point your LSP client at `./cure-lsp`. Neovim users:

```lua
require('lspconfig.configs').cure = {
  default_config = {
    cmd = { '/path/to/cure/cure-lsp' },
    filetypes = { 'cure' },
    root_dir = function(fname)
      return vim.fs.dirname(fname)
    end,
  },
}
require('lspconfig').cure.setup({})
```

## What Comes Next

The foundation is solid. The areas that will see the most growth:

- **Deeper dependent types**: full value-parameter tracking through function
  calls, type-level arithmetic verification, length-indexed vectors
- **Cross-module protocol dispatch**: ETS-backed global protocol registry
- **LSP code actions**: add missing match arms, fix imports, rename symbols
- **SMT counterexample extraction**: when a proof fails, show the specific
  values that break it
