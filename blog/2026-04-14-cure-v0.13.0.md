# Cure v0.13.0: The Type System Grows Teeth

v0.12 gave Cure a body. v0.13 gives it teeth.

The headline: when you write `fn safe_divide(a: Int, b: Int) -> Int when b != 0`,
the compiler now *actually checks that constraint at every call site*. Call
`safe_divide(10, 0)` and Z3 proves the violation before a single BEAM
instruction runs. Call `safe_divide(10, x)` where `x` came from user input
and the compiler tells you it cannot prove the constraint -- your job to
handle it.

This is not a runtime check. This is a compile-time proof.

## What changed

Seven things, each deepening a subsystem that v0.12 established.

### 1. Dependent type verification

Functions with `when` guards now register as constrained types:

```cure
fn safe_divide(a: Int, b: Int) -> Int when b != 0 = a / b
fn positive_double(x: Int) -> Int when x > 0 = x * 2
```

At call sites, the compiler substitutes literal arguments into the guard
predicate and sends it to Z3. If the predicate is unsatisfiable after
substitution, you get a warning:

```
warning: call to 'safe_divide': guard constraint may be violated
```

When the proof fails, the solver extracts a counterexample from Z3's
model output -- not just "it failed" but "it failed because x = -1."

Along with this comes `Std.Vector`, the first stdlib module that uses
dependent types: length-indexed vectors where `cons` increments the
length and `append` sums them.

### 2. Protocols that cross module boundaries

In v0.12, `impl Show for Int` in module A was invisible to module B.
Now every `impl` is registered in a global ETS table during compilation:

```elixir
ProtocolRegistry.lookup_impl("Show", "show", "Int")
# => {:ok, :std_show}
```

When you compile `Std.Show`, the registry learns that `:std_show` provides
`show/1` for Int, Float, String, Bool, and Atom. When you compile `Std.Eq`,
it learns about `eq/2`. This is the foundation for cross-module dispatch --
the registry knows who implements what, even across separate compilation
units.

### 3. An LSP that does more than highlight errors

The Language Server gained three new capabilities:

**Go-to-definition.** Hover over a function call, jump to where it is
defined. Simple text search for now -- no symbol table indexing -- but it
works for same-file definitions and that covers the majority of editing.

**Document symbols.** The outline view in your editor shows modules,
functions, types, and protocols in a hierarchical tree. Parsed from the
AST, so it works even when the file has type errors.

**Code actions.** Two quick fixes:

- Non-exhaustive match warning? The LSP offers "Add wildcard pattern."
- Unbound variable? If it is within Levenshtein distance 2 of a keyword,
  the LSP offers "Did you mean '...'?"

### 4. The optimizer learns to inline

The optimizer pipeline grew from 3 passes to 5:

1. **Inline** -- the new pass. Finds small, pure, non-recursive functions
   (body <= 5 AST nodes, no side effects) and substitutes their bodies
   at call sites. `fn double(x) = x * 2` called as `double(21)` becomes
   `21 * 2`, which the constant folder then reduces to `42`.

2. Constant fold (unchanged)
3. Dead code elimination (unchanged)
4. Pipe inline (unchanged)

5. **Guard simplify** -- the other new pass. Applies algebraic rules:
   `P and P` becomes `P`. `true and P` becomes `P`. `not not P` becomes
   `P`. `false or P` becomes `P`. These arise naturally in multi-clause
   functions where guard coverage analysis generates redundant conjunctions.

### 5. FSMs that do things on transitions

The FSM compiler now handles the `when` and `do` keywords on transitions.
The parser already supported them; v0.13 compiles them:

```cure
fsm Counter
  Counting --tick when count > 0 do count = count - 1--> Counting
  Counting --tick when count == 0--> Done
```

Guards become Erlang guard expressions on the `handle_event` clause.
Actions become data-mutating expressions in the clause body. The state
data map flows through transitions and comes out modified.

The runtime also gained `health_check/1`: a single call that returns
whether the FSM is alive, what state it is in, how many events it has
processed, how long it has been running, and what the last event was.

### 6. Errors that show you the code

The CLI now displays the offending source line with a caret when
compilation fails:

```
error: type mismatch
 --> hello.cure:3
  | function 'bad' declared return type String but body has type Int
  |   fn bad() -> String = 42
  |                        ^
```

And there is now a structured error catalog. Each error type has a code:

- E001: Type Mismatch
- E002: Unbound Variable
- E003: Arity Mismatch
- E004: Non-Exhaustive Match
- E005: Constraint Violation

Run `cure explain E001` to see a description, example, and fix suggestion.

### 7. Four more stdlib modules

The standard library grew from 13 to 17 modules:

- **Std.Map** -- 14 functions wrapping Erlang's `:maps` module. `new`,
  `put`, `get`, `get_or`, `remove`, `has_key`, `size`, `keys`, `values`,
  `to_list`, `from_list`, `merge`.

- **Std.Set** -- 11 functions built on Std.Map. Sets are maps with `:true`
  values. `union`, `intersection`, `difference` work as expected.

- **Std.Option** -- 10 functions extracted from Std.Core into a dedicated
  module. Adds `filter` and `zip` that Core did not have.

- **Std.Functor** -- the `Functor` protocol with `fmap/2`. One impl so
  far (List via `Std.List.map`). The pattern is set for containers to
  follow.

Total: 17 modules, roughly 200 exported functions, all self-hosted in Cure.

606 tests. Zero credo issues. Zero compilation warnings. 48 Elixir source

The entire v0.13 implementation -- seven phases from planning to tagged
release -- was done in a single session.

## Getting started

```bash
git clone https://github.com/am-kantox/cure.git
cd cure
mix deps.get && mix test
mix escript.build
./cure version          # Cure 0.13.0
./cure run examples/dependent_types.cure
./cure explain E001
```
