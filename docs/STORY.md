# `cure story` -- narrative architecture generator (v0.32.0)

`cure story` reads a project's `.cure` source tree, builds a structural
outline of its runtime components, and emits a readable `STORY.md` that
introduces the system top-down:

    apps -> supervisors -> actors -> FSMs -> types

The document is generated from the actual source, so it stays accurate
as the project evolves. Run `cure story` again after any structural
change to refresh it.

## Usage

```bash
# Write STORY.md in the current project root
mix cure.story

# Custom output path
mix cure.story --out docs/ARCHITECTURE.md

# Print to stdout without writing a file
mix cure.story --stdout

# Include Mermaid state diagrams for each FSM
mix cure.story --diagrams

# Generate an HTML version
mix cure.story --format html --out docs/story.html

# Override the project name in the document title
mix cure.story --project-name "Cure Atelier"
```

## How the outline is built

`cure story` parses every `.cure` file under `lib/` with the same
lightweight parser pass used by `cure doc`. Files that fail to parse are
silently skipped.

Container types are classified as follows:

| Cure keyword | Section in STORY.md |
|-------------|---------------------|
| `app`       | Applications |
| `sup`       | Supervisors |
| `actor`     | Actors |
| `fsm`       | FSMs |
| `rec`, `type`, `enum` | Shared Types |

Within each section, nodes appear in source-file alphabetical order.

## Document structure

A generated `STORY.md` looks like this:

```markdown
# my_project

This document describes the architecture of the `my_project` system,
generated by `cure story` from the project source.

## Overview

The system comprises 1 application(s), 2 supervisor(s), 3 actor(s),
1 FSM(s), 5 shared type(s).

## Applications

### `Demo`

The `Demo` application is the root of the system. It starts
supervisor(s): `Demo.Root`.

## Supervisors

### `Demo.Root`

The `Demo.Root` supervisor applies a `one-for-one` restart strategy.
It manages: `Demo.Worker` and `Demo.Logger`.

## Actors

### `Demo.Worker`

The `Demo.Worker` actor declares effect(s): Io. It handles 2 message
type(s): `Request` and `Response`.

## FSMs

### `Turnstile`

The `Turnstile` FSM models 2 state(s): `Locked`, `Unlocked`.
3 transition(s) are defined.

## Shared Types

- **`Ticket`** (record)
- **`Status`** (enumeration (ADT)) -- access control status
```

## Mermaid FSM diagrams

Pass `--diagrams` to embed a `stateDiagram-v2` Mermaid block for each
FSM section:

    ```mermaid
    stateDiagram-v2
      title Turnstile
      Locked -->|coin| Unlocked
      Unlocked -->|push| Locked
    ```

Mermaid diagrams render automatically in GitHub Markdown, GitLab, and
any Mermaid-aware viewer.

## HTML output

`--format html` wraps the Markdown in a minimal HTML shell suitable for
standalone browser viewing:

```bash
mix cure.story --format html --out docs/story.html
open docs/story.html
```

## Keeping STORY.md up to date

Add a step to your CI pipeline that regenerates `STORY.md` and fails if
the result differs from the committed version:

```bash
mix cure.story --out /tmp/STORY.md
diff STORY.md /tmp/STORY.md || (echo "STORY.md is out of date" && exit 1)
```

Or commit the generation step as a pre-commit hook:

```bash
#!/bin/sh
mix cure.story
git add STORY.md
```
